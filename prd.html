Executive Summary

Qwality Caps is a production-ready, full-stack e-commerce platform designed to sell premium artisan headwear online. The platform enables a seamless end-to-end shopping experience — from product discovery to payment — while giving the brand owner complete operational control through a powerful admin dashboard.

The platform is live at qwality-frontend.vercel.app, backed by a secure REST API at qwality-backend.onrender.com, and powered by MongoDB Atlas in the cloud.

Product Vision: To build a premium, conversion-focused online storefront that matches the artisan quality of the Qwality Caps brand — fast, beautiful, and trustworthy — enabling the business to operate entirely online with zero third-party marketplace dependency.

1.1 Scope Summary

Area Description
Storefront Customer-facing shop, product pages, cart, checkout, order history
Admin Panel Full backend management — products, orders, users, coupons, CMS
Auth System Registration, login, email verification, password reset
Payments Razorpay gateway integration + Cash on Delivery
Media Cloudinary-powered image upload and delivery
Communication Transactional emails (verification, password reset)

Problem Statement & Opportunity

2.1 The Problem

Artisan cap brands traditionally rely on physical retail, social media DMs, or third-party marketplaces (Amazon, Etsy) to sell their products. This approach creates several pain points:

Pain Point Impact
No owned online presence Brand identity diluted on third-party platforms; high commission fees (8–15%)
Manual order management Orders tracked on spreadsheets or WhatsApp — error-prone and unscalable
No customer data ownership Can't re-market, build loyalty programs, or track purchase history
Limited product showcasing No variant selection, no gallery, no reviews — poor buyer confidence
Cash-only / informal payments No digital payment option reduces conversion, especially from younger buyers

2.2 The Opportunity

The D2C (Direct-to-Consumer) e-commerce market in India is growing at ~40% YoY. Artisan and handcrafted goods are seeing a resurgence in demand driven by social media discovery. A well-built, mobile-friendly, SEO-optimized storefront gives Qwality Caps the opportunity to capture this demand directly — without marketplace dependency.

2.3 Value Proposition

Stakeholder Value Delivered
Business Owner (Admin) Full control — manage products, pricing, stock, orders, promotions and content without any technical knowledge
Customer A premium, fast, secure shopping experience with multiple payment options, order tracking and account management
Business Zero commission on sales, owned customer data, brand credibility, repeat purchase enablement via accounts

Non-Functional Requirements

8.1 Performance

Requirement Target Priority
Page load time (homepage, shop) < 3 seconds on 4G mobile Must
API response time (P95) < 200ms for read endpoints Must
Image delivery Cloudinary CDN with auto format/quality Must
Checkout latency Order placement < 3s end-to-end Must

8.2 Security

Requirement Implementation Priority
Password security bcrypt hashing (10 rounds) Must
XSS protection httpOnly cookies + Helmet CSP Must
CSRF protection SameSite cookie policy Must
API abuse prevention Rate limiting: 100 req/15 min/IP Must
Input sanitisation express-validator on all POST/PUT Must
HTTPS only Enforced via Vercel + Render + secure cookies Must
Secret management Environment variables only, never in code Must

8.3 Reliability

Requirement Target
API uptime 99.9% (Render + MongoDB Atlas SLA)
Database resilience MongoDB Atlas 3-node replica set with auto-failover
Error handling All errors caught and returned in consistent JSON format
Server crash recovery Render auto-restarts on crash; process.exit(1) on DB failure

8.4 Usability

Requirement Standard
Mobile responsiveness All pages fully responsive on 320px–1440px viewport
Accessibility Semantic HTML, proper heading hierarchy, ARIA labels on interactive elements
Browser support Latest 2 versions of Chrome, Firefox, Safari, Edge
Form validation Inline real-time validation with clear error messages
Loading states Spinners/skeletons on all async operations

8.5 Scalability

Requirement Approach
Concurrent users Stateless JWT auth allows horizontal backend scaling
Database growth MongoDB Atlas scales vertically/horizontally on demand
Media storage Cloudinary scales infinitely; no local disk dependency
Frontend scaling Vercel Edge CDN serves globally with zero config

Constraints & Assumptions

9.1 Technical Constraints

Constraint Impact
Frontend and backend on different domains (Vercel + Render) Requires SameSite=None; Secure cookies and strict CORS origin whitelist
Render free tier cold starts (~30–50s) First request after inactivity may be slow; upgrade to paid tier for production
MongoDB Atlas free tier (512MB storage) Sufficient for early stage; upgrade as catalog grows
Cloudinary free tier (25GB bandwidth/month) Monitor usage; upgrade if media-heavy
Image upload max 5MB per file Admin must compress images before upload or use Cloudinary transformations
Razorpay test mode in development Use live keys for production payments

9.2 Business Constraints

Constraint Detail
Single admin user initially The platform supports one admin role; multi-admin with granular permissions is a future scope
No physical store integration Inventory is managed manually in the admin dashboard
India-focused initially Currency is INR; shipping logic is flat fee; international shipping not in scope
No third-party logistics (3PL) integration Shipping is managed manually; no auto label generation

9.3 Assumptions

  • The admin will manually update order statuses upon shipping
  • Product images will be prepared and optimised before upload by the admin
  • All transactions will be in Indian Rupees (INR)
  • Email delivery depends on the SMTP provider configured — production requires a reliable SMTP (not Mailtrap sandbox)
  • Razorpay live keys must be configured in Render environment for real payments
  • MongoDB Atlas network access is configured to allow Render's IP range

Out of Scope

The following features are explicitly not included in the current version (v1.0) of Qwality Caps:

Feature Reason
Native mobile app (iOS / Android) Web-first approach; mobile app planned for Phase 3
Social login (Google, Apple, Facebook) Phase 2 scope
Automated logistics / shipment tracking API Manual order status updates sufficient at current scale
Multi-vendor / marketplace model Single-brand store only
Subscription / recurring payments Not applicable to current product model
International shipping & multi-currency India market only in v1
Live chat / customer support widget Phase 2 scope
Product comparison feature Future scope
Abandoned cart email recovery Phase 2 scope
Inventory purchase orders Out of scope for digital platform

Risks & Mitigations
Risk Severity Likelihood Mitigation
Render backend cold start delays first requests High High Upgrade to Render paid plan ($7/mo) to keep instance always-on
Cross-site cookie blocked by browser updates High Medium SameSite=None + Secure already implemented; fallback to Authorization header supported
Razorpay test keys used in production High Medium Replace RAZORPAY_KEY_ID and KEY_SECRET with live keys before going live
Email delivery failure (SMTP sandbox in prod) Medium High Replace Mailtrap with production SMTP (SendGrid, Resend, or Gmail SMTP)
MongoDB Atlas storage limit hit (512MB free) Medium Low Monitor Atlas dashboard; upgrade to M10 paid cluster when approaching limit
Cloudinary bandwidth exceeded (free tier) Medium Low Monitor usage; compress images; upgrade plan if needed
Admin credentials compromised High Low Strong password enforced; JWT expiry 1 day; bcrypt hashing; rate limiting on login
Payment fraud or duplicate orders High Low Razorpay signature verification server-side before order confirmation
