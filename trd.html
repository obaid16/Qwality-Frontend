2.1 Request Lifecycle

  1. User interacts with the Next.js frontend (Vercel)
  2. Frontend calls apiFetch() utility → sends HTTP request with credentials: "include"
  3. Request arrives at Render → passes through Morgan logging → Rate Limiter → CORS middleware → Helmet security headers
  4. JWT middleware verifies accessToken httpOnly cookie (or Authorization header)
  5. Route handler → Service layer → Repository layer → MongoDB Atlas
  6. Response returned as { success, data, message } JSON envelope

Backend — API Specification

All API routes are prefixed with /api/v1. Responses follow a consistent JSON envelope:

{ "success": true, "statusCode": 200, "data": { ... }, "message": "..." }
Authentication: Protected routes require a valid accessToken sent as an httpOnly cookie or via Authorization: Bearer <token> header.

06
Authentication & Security

6.1 Auth Flow

  1. User submits credentials → POST /api/v1/auth/login
  2. Server verifies password with bcrypt.compare()
  3. Server generates Access Token (JWT, 1 day expiry) and Refresh Token (JWT, 10 days)
  4. Both tokens set as httpOnly; Secure; SameSite=None cookies (cross-site safe)
  5. Subsequent requests automatically include cookies → verifyJWT middleware decodes token
  6. On token expiry → client calls POST /auth/refresh-token to get new pair
  7. On logout → cookies cleared + refresh token nulled in DB

6.2 Cookie Configuration

const cookieOptions = { httpOnly: true, // Not accessible via JS (XSS protection) secure: process.env.NODE_ENV === "production", // HTTPS only in production sameSite: process.env.NODE_ENV === "production" // cross-site safe for Vercel→Render ? "none" : "strict", };

7.3 API Client

All API calls go through src/utils/api.js — a centralized fetch wrapper:

  • Reads base URL from NEXT_PUBLIC_API_URL env var
  • Automatically includes credentials: "include" for cookie auth
  • Handles FormData vs JSON content-type automatically
  • Parses unified error response format
  • Auto-redirects to /auth on 401 (outside of auth pages)

10.4 Cross-Origin Configuration Summary

Production Cross-Site Setup:
Frontend (vercel.app) and Backend (onrender.com) are on different domains.

Required settings for cookies to work cross-site:
  • Backend: CORS_ORIGIN=https://qwality-frontend.vercel.app (no trailing slash)
  • Backend: NODE_ENV=production (enables sameSite: "none" + secure: true)
  • Frontend: NEXT_PUBLIC_API_URL=https://qwality-backend.onrender.com/api/v1
  • API client: credentials: "include" on all fetch calls
